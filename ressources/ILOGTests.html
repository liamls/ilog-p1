1. Tests unitaires

COURS

Modélisation

Les cas d'utilisation :

  • permettent de découvrir les classes et leurs méthodes publiques ;
  • mettent en jeu plusieurs classes dans différentes circonstances ;

Un générateur de code peut créer des ébauches de code.

Faute d'outils de modélisation ou en complément, on peut coder les cas d'utilisation :

  • pour affiner les classes et leurs méthodes publiques ;
  • créer les ébauches des classes ;
  • d'assurer une première cohérence grâce au compilateur.
Penser à la façon dont on voudrait utiliser les objets à concevoir.
  • Se mettre à la place du programmeur qui va les utiliser :
    • quelles classes ?
    • de quelles méthodes voudrait-il disposer ?
  • Définir éventuellement des interfaces qu'une classe doit réaliser.
  • Une interface formalise les besoins d'un objet utilisateur :
    • autre objet utilisateur,
    • autres besoins,
    • autres méthodes,
    • autre interface.

Phase de spécification

Le "quoi" (spécification) avant le "comment" (implémentation) !

À l'issue de la phase de modélisation, tout n'est pas complètement spécifié.

La rédaction des tests et de la JavaDoc participe à l'effort de spécification.

  • On rédige conjointement :
    • un scénario de test unitaire ;
    • une ébauche de la classe à tester avec des méthodes publiques "vides" ;
    • la JavaDoc de la classe testée et de ses méthodes publiques ;
  • les trois sont indissociables ;
  • les fonctions de l'ébauche retournent 0, null ou false.

Rédiger la JavaDoc et les tests unitaires suppose qu'on ait des idées sur :

  • les services offerts par un ou plusieurs objets qui collaborent ;
  • la place des objets dans le contexte où ils seront utilisés ;
  • on imagine la classe à concevoir telle qu'on voudrait l'utiliser.

NB JavaDoc et tests unitaires se complètent :

  • rédiger la JavaDoc :
    • pousse à réfléchir à ce qu'on conçoit ;
    • met à l'épreuve les idées qu'on se fait des futurs objets ;
  • les tests :
    • précisent sur des cas concrets le fonctionnement attendu ;
    • mettent un peu plus à l'épreuve les idées qu'on se fait des futurs objets ;
    • doivent vérifier les affirmations de la JavaDoc.
  • Une réflexion précoce sur les cas limites :
    • est structurante ;
    • conduit généralement in fine à un code plus maintenable.

NB Limites des tests unitaires

Ils sont :

  • capables de révéler la présence de bugs ;
  • incapables de garantir leur absence.

NB Lest tests unitaires sont une composante des approches agiles :

  • on procède par itérations ;
  • on développe la fonctionnalité principale en premier ;
  • le codage est guidé par les tests.

Phase d'implémentation

  • les tests intégrés au build automatisé (make, ant... intégration continue) ;
  • l'implémentation s'arrête quand les tests passent OK ;
  • on ne développe que ce qui est nécessaire ;
  • les tests guident le versioning : on archive chaque avancée dans l'implémentation.

Maintenance corrective

  • on écrit de nouveaux tests qui révèlent le bug ;
  • quand on cherche à corriger le bug :
    • les tests existants détectent plus facilement la régression ;
    • les nouveaux tests passent OK ;
  • on archive les tests et le code corrigé.

Maintenance évolutive

  • les tests donnent confiance ;
  • pour refactoriser ;
  • pour ajouter de nouvelles fonctionnalités.

---

JUnit
  • est un outil écrit en Java pour faciliter la création de scénarii de tests unitaires ;
  • est composé des fichiers junit4.jar et hamcrest.core.jar.
  • intégré à eclipse ;
  • aussi utilisable indépendamment.

Annotations

  • Test, Before et After ne sont ni des classes ni des interfaces mais des annotations ;
  • une annotation peut cibler une classe, un champ, une méthode (c'est le cas ici), un paramètre ;
  • une annotation peut (ou non) être "runtime" et se retrouver (ou non) dans le .class ;
  • une annotation runtime peut être découverte par introspection ;
  • les méthodes annotées par @Test sont des méthodes de test ;
  • le champ annoté par @Rule caractérise des objets de JUnit :
    • ici TestName est utilisé dans les méthodes pour obtenir leur nom ;
  • les méthodes annotées par @Before sont exécutées avant chaque méthode de test :
    • mise en place de l'environnement de test surtout s'il est complexe ;
  • les méthodes @After sont exécutées après chaque méthode de test :
    • recyclage de l'environnement de test.

JUnitCore

  • est le "test runner" (dérouleur de scénario) de JUnit4 ;
  • est une application qui prend en argument le nom qualifié d'une classe de test
java -cp bin:bin-tests:lib/junit4.jar:lib/hamcrest.core.jar org.junit.runner.JUnitCore fr.imtld.ilog.RunnerTest
  • utilise un chargeur de classes pour charger en mémoire la classe de test ;
  • découvre par introspection les méthodes annotées par @Test, @Before ou @After ;
  • découvre par introspection un constructeur sans arguments ;
  • crée une instance de la classe de test de manière réflexive ;
  • appelle de manière réflexive :
    • les méthodes annotées par Before ;
    • une méthode annotée par Test ;
    • les méthodes annotées par After ;
  • recommence avec une autre méthode annotée par Test...

2.1 Contexte

Fifo SDL

On se propose de tester une classe Fifo particulière ;

  • elle est utilisée dans la réalisation de modèles SDL ;
  • SDL = Specification and Description Language ;
  • un modèle SDL spécifie le comportement de processus ;
  • les processus communiquent par signaux.

Chaque instance de processus possède une instance de Fifo ;

  • d'autres instances de processus peuvent y déposer des signaux à son attention ;
  • les signaux du modèle SDL sont réalisés à l'aide de petites classes ;
  • comme CR (Connection Request) ou CC (Connection Confirmation).

Un processus émetteur dépose un signal à la queue de la Fifo :

_fifo.add(signal);

Le processus destinataire :

  • teste le nombre de signaux présents dans la fifo avec getSize() ;
  • acquiert le signal en tête de fifo avec getHead() ;
  • puis le retire avec remove().
if (_fifo.getSize() > 0) {
	Object o = _fifo.getHead();
	_fifo.remove();

Mise en réserve

Selon l'état où il se trouve,

  • le processus destinataire peut ignorer certains signaux en tête de fifo.
  • Les signaux ne sont pas retirés de la fifo ;
  • mais rendus inaccessibles en décalant la position de la tête de la fifo.

Exemple

  • On ignore les signaux de type CR :

_fifo.save(CR.class);

  • si initialement la tête de la fifo est en position 0 :

  • la position de la tête de fifo se décale à la position 2 ;
  • le signal DT se retrouve en tête de fifo...

  • d'où il peut être utilisé puis retiré ;
  • la fifo est alors considérée comme vide :

  • la mise en réserve des signaux peut être annulée :

_fifo.save(null);

  • la tête de fifo retrouve sa position normale en position 0 ;
  • les signaux CR se retrouvent en tête (où ils pourront être utilisés) :

Ainsi, un modèle SDL peut facilement être réalisé grâce à cette fifo spéciale.

2.2 Démarche

On détaille la démarche de conception :

Démarche de conception

On spécifie avant de coder : le "quoi" avant le "comment".

  1. Identifier les objets utilisateurs de l'objet à concevoir et identifier leurs besoins ;
  2. spécifier chaque type de besoin au moyen d'une interface que l'on documente ;
  3. créer un objet réalisant les interfaces par des méthodes vides ;
  4. écrire les scénarios de test de façon à décrire le comportement attendu de l'objet ;
  5. coder les méthodes de l'objet de façon à ce que le test passe ;
  6. archiver à chaque progression.

Dans ce TP, les phases 1 à 4 sont déjà réalisées. On a identifié :

  • une interface FifoQueue pour le processus producteur ;
  • l'interface FifoHead pour le processus consommateur.
En pratique, l'objet créé à la phase 3 n'est pas fourni. Il faudra :
  • le recréer ;
  • importer les commentaires JavaDoc des interfaces.

package fr.imtld.ilog;

import org.junit.Before;
import org.junit.Rule;
import org.junit.Test;
import org.junit.rules.TestName;
import org.junit.runner.JUnitCore;

import static org.junit.Assert.*;

/**
 * JUnit test for fr.imtld.ilog.Fifo
 * 
 * @version for JUnit4
 * @author Christophe TOMBELLE
 */
public class FifoTest {
	protected Double o3, o4, o5;
	protected Fifo fifo;
	protected FifoHead fifoHead;
	protected FifoQueue fifoQueue;
	@Rule
	public TestName namTest = new TestName();

	/**
	 * Build objects useful to the test.
	 */
	@Before
	public void setUp() {
		o3 = 3.;
		o4 = 4.;
		o5 = 5.;
		fifo = new Fifo();
		fifoHead = fifo;
		fifoQueue = fifo;
		System.out.println(namTest.getMethodName());
	}

	// helper methods

	/**
	 * Check the Fifo state : size and head signal.
	 * 
	 * @param size    The expected size.
	 * @param headExp The expected signa.
	 */
	protected void checkState(int size, Object headExp) {
		assertEquals(size, fifo.getSize());
		Object head = fifo.getHead();
		if (headExp instanceof Integer)
			assertEquals(headExp, head);
		else
			assertSame(headExp, head);
	}

	/**
	 * Fill the Fifo with the specified signals.
	 * 
	 * @param signals The signals.
	 */
	protected void fill(Object... signals) {
		for (int i = 0; i < signals.length; i++)
			fifoQueue.add(signals[i]);
	}

	/**
	 * Remove n signals from the Fifo.
	 * 
	 * @param n The number of signals to remove.
	 */
	protected void remove(int n) {
		for (int i = 0; i < n; i++)
			fifoHead.remove();
	}

	// test methods
	@Test
	public void testInitiallyEmpty() {
		checkState(0, null);
	}

	@Test
	public void testRemoveEmptyFifo() {
		fifoHead.remove();
	}

	@Test
	public void testSaveEmptyFifo() {
		fifoHead.save(Double.class);
		fifoHead.save(4);
	}

	@Test
	public void testAddNull() {
		fifoQueue.add(null);
		assertEquals(0, fifoHead.getSize());
	}

	@Test
	public void testAddObjects() {
		fill(o5, o4, o3);
		checkState(3, o5);
	}

	@Test
	public void testAddInts() {
		fill(5, 4, 3);
		checkState(3, 5);
	}

	@Test
	public void testRemoveObject() {
		fill(o5, o4, o3);
		remove(1);
		checkState(2, o4);
	}

	@Test
	public void testRemoveInt() {
		fill(5, 4, 3);
		remove(1);
		checkState(2, 4);
	}

	@Test
	public void testAddRemove() {
		fill(5, o4, 3);
		remove(3);
		checkState(0, null);
	}

	@Test
	public void testSaveObjects() {
		fill(o5, o4, o3, 4, 5, 5);
		fifoHead.save(Double.class);
		checkState(3, 4);
	}

	@Test
	public void testSaveInts() {
		fill(3, 3, 3, o3, o4, o5);
		fifoHead.save(3);
		checkState(3, o3);
	}

	@Test
	public void testAllSavedObjects() {
		fill(o5, o4, o3);
		fifoHead.save(Double.class);
		checkState(0, null);
	}

	@Test
	public void testAllSavedInts() {
		fill(3, 3, 3);
		fifoHead.save(3);
		checkState(0, null);
	}

	@Test
	public void testSizeResetSaved() {
		fill(o5, o4, o3, 4, 5, 5);
		fifoHead.save(Double.class);
		// reset savings
		fifoHead.save(null);
		checkState(6, o5);
	}

	/**
	 * This test case has a main so that it can be launched as a standalone
	 * application.
	 * 
	 * @param straArgs The command line arguments (unused).
	 */
	public static void main(String[] straArgs) {
		JUnitCore.main("fr.imtld.ilog.FifoTest");
	}
}

Java : connaître le type réel d'un objet

On rappelle la différence entre :

  • le type déclaratif (ou statique) d'un objet (selon lequel il est déclaré)
  • et son type dynamique (ou réel, selon lequel il est instancié).
Fifo fifo = new Fifo();
FifoHead fifoHead = fifo;

NB

  • 1ère ligne : types déclaratif et dynamique identiques (Fifo) ;
  • 2ème ligne : type déclaratif FifoHead / type dynamique Fifo.

Les objets stockés dans un tableau ou une liste peuvent être de type quelconque. Ainsi, le résultat de la méthode get est de type déclaratif Object. 

Object o = _list.get( 0 );

Son type dynamique peut être, par exemple, un Double ou un Integer. :

_list.add ( 5.0 );	// autoboxing en new Double(5.0)
_list.add ( 5 );	// autoboxing en new Integer(5)
On peut vérifier que l'objet o est bien d'un type désigné par une instance de la classe Class : 
public void verifier( Class cls ) {
    Object o = _list.get(0);
    if (o.getClass() == cls.isInstance(o)
        ...
}

NB

  • Le code de verifier n'utilise aucune connaissance a priori du type de o.
  • C'est ce qu'on attend de la méthode Fifo#save(Class cls).

	protected long _lT0 = 0;

	protected long now() {
		return System.currentTimeMillis();
	}

	public Clock() {
		_lT0 = now();
	}

	public int getDay() {
		return (int) (now() - _lT0) / 86400000;
	}

	public int getHour() {
		return (((int) (now() - _lT0)) / 3600000) % 24;
	}

	public int getMinute() {
		return (((int) (now() - _lT0)) / 60000) % 60;
	}

	public int getSecond() {
		return (((int) (now() - _lT0)) / 1000) % 60;
	}

	public int getTime() {
		return (int) (now() - _lT0) % 604800000;
	}

	public void setDay(int iDay) {
		if ((((int) (now() - _lT0)) / 86400000) < iDay) {
			_lT0 = (long) (_lT0 - (iDay - (((int) (now() - _lT0)) / 86400000)) * 86400000);
		} else {
			_lT0 = (long) (_lT0 - ((((int) (now() - _lT0)) / 86400000) - iDay) * 86400000);
		}
	}

	public void setHour(int iHour) {
		if (((((int) (now() - _lT0)) / 3600000) % 24) < iHour) {
			_lT0 = (long) (_lT0 - (iHour - ((((int) (now() - _lT0)) / 3600000) % 24)) * 3600000);
		} else {
			_lT0 = (long) (_lT0 - (((((int) (now() - _lT0)) / 3600000) % 24) - iHour) * 3600000);
		}
	}

	public void setMinute(int iMinute) {
		if (((((int) (now() - _lT0)) / 60000) % 60) < iMinute) {
			_lT0 = (long) (_lT0 - (iMinute - ((((int) (now() - _lT0)) / 60000) % 60)) * 60000);

			_lT0 = (long) (_lT0 - (((((int) (now() - _lT0)) / 60000) % 60) - iMinute) * 60000);

3.1 Problématique du test en isolation

  • La classe Fifo se teste facilement car elle ne dépend d'aucun autre objet si ce n'est un objet liste réputé déjà testé.
  • Tout n'est pas toujours si simple.
  • Si la classe testée collabore avec une ou plusieurs autres classes, elle dépend de ces classes partenaires ;
  • le test n'est plus unitaire :
    • les classes partenaires doivent être testées et implémentées avant la classe qu'on voulait tester initialement ;
    • la dépendance peut être bidirectionnelle : les classes doivent alors être testées et implémentées conjointement ;
    • une classe partenaire peut elle-même collaborer avec une autre classe donc dépendre de cette autre classe ;
    • cette chaîne de collaboration et de dépendances empêche le développement en parallèle par plusieurs développeurs ;
  • Dans un test dit unitaire, on cherche à tester une classe à la fois, même si elle a des classes partenaires.
  • Lors du test, les objets partenaires sont remplacés par des objets dits mocks ("bidons", imités) ;
  • en dehors du test, les objets partenaires sont les vrais objets.
L'objet mock : "Yes, I'm the great pretender...

...
I seem to be what I'm not (you see)"
[Buck RAM 1955]

La technique des objets mocks isole l'objet testé :

  • Un objet mock imite l'objet qu'il remplace :
    • il présente la même interface ;
    • il a le même comportement apparent.
  • Mais en fait, son comportement est "bidonné" (avec la complicité du scénario de test) :
    • le comportement n'est bon que dans les circonstances du test ;
    • le mock est piloté par le scénario de test via des méthodes supplémentaires ;
    • les méthodes de pilotage sont invisibles à l'objet testé.
  • un mock ne doit contenir aucune logique applicative.

Cependant, on doit renoncer au mocking :

  • s'il est plus simple de rédiger la logique applicative ;
  • que d'obscurcir le test en cherchant absolument à respecter ce principe.

On teste alors :

  1. la dépendance pour l'implémenter ;
  2. la classe qui en dépend.

Testabilité

Il peut être nécessaire une classe pour la rendre testable. La classe testée :

  • doit permettre le remplacement des objets réels par des objets mocks ;
  • ne  doit pas imposer une implémentation particulière (mock ou réelle) du service ;

Par exemple, si elle utilise System.currentTimeMillis(), elle :

  • impose une implémentation réelle d'un service de mesure du temps écoulé ;
  • verrouille tout remplacement par une implémentation mock de ce service.

Si au contraire, elle ne fait référence qu'à une interface comme ITimeBase, elle ouvre la possibilité :

  • d'une implémentation mock (classe TimeBaseMock) ;
  • d'une implémentation réelle (classe TimeBase).

Instantiation

Une ITimeBase (type déclaratif) désigne une TimeBaseMock ou une TimeBase (types dynamiques) en fonction de l'instantiation qui a eu lieu :

  • ITimeBase mock = new TimeBaseMock();
  • ITimeBase time = new TimeBase();

Il est donc important que l'instantiation n'ait pas lieu dans la classe testée. On peut utiliser :

  1. soit une fabrique : ITimeBase time = factory.getTimeBase(); dans la classe testée ;
  2. soit l'injection de dépendances (dans la classe de test) :
    • ITimeBase mock = new TimeBaseMock();
    • Clock tested = new Clock(mock);

NB

  1. Le test substituera à la fabrique réelle une fabrique de test (getTimeBase() fournissant une TimeBaseMock) ;
  2. l'application réelle injectera la TimeBase réelle : Clock clock = new Clock(new TimeBase());.

Les objets testés doivent être isolés des dépendances :

  • applicatives ;
  • envers le système d'exploitation (comme un fichier) ;
  • envers un environnement d'exécution :
    • base de données ;
    • conteneur J2EE.

Exemple

 Clock, l'objet testé, offre un service de calendrier hebdomadaire :

  1. Il dépend de l'OS : System.currentTimeMillis() donne l'instant présent ;
  2. il est hors de question qu'un test pertinent dure une semaine.

On remplace :

  • la dépendance envers l'OS ;
  • par une dépendance envers une interface (e.g. ITimeBase).

NB

  • Dans le scénario de test, l'objet réalisant l'interface sera un objet mock (e.g. TimeBaseMock) :
    • fournissant un instant présent simulé ;
    • piloté par le test (setNow).
  • Dans l'application, l'objet réalisant l'interface sera un objet applicatif (e.g. TimeBase) :
    • déléguant à System.currentTimeMillis() ;
    • sans valeur ajoutée (inutile de tester).
  • L'objet, mock ou applicatif, est :
    • instancié en dehors de l'objet testé ;
    • injecté dans l'objet testé ;
    • déclaré de type ITimeBase dans la classe Clock.

package fr.imtld.ilog;

public class Counter implements ICounter {
	protected IThumbWheel[] wheels;

	public Counter() {
		IFactory fact = Factory.get();
		wheels = fact.createThumbWheels(4);
		for (int i = 0; i < wheels.length; i++)
			wheels[i] = fact.createThumbWheel();
	}

	@Override
	public void clear() {
		int len = wheels.length;
		for (int i = 0; i < len; i++)
			while (wheels[i].value() != 0)
				wheels[i].up();
	}

	@Override
	public void up() {
		int len = wheels.length;
		int i = 0;
		while (i < len) {
			IThumbWheel wheel = wheels[i++];
			wheel.up();
			if (wheel.value() != 0)
				break;
		}
	}

	@Override
	public int getValue() {
		int len = wheels.length;
		int val = 0;
		for (int i = len - 1; i >= 0; i--)
			val = 10 * val + wheels[i].value();
		return val;
	}

	@Override
	public String toString() {
		return Integer.toString(getValue());
	}
}

		return iUp;
	}

	@Override
	public int value() {
		return values[i++];
	}

	@Override
	public void up() {
		iUp++;
	}
}

NB

  • en dehors d'une démarche pédagogique, la "mocker" n'a donc pas d'intérêt ;
  • le constructeur spécifie les valeurs consécutives que value() doit retourner ;
  • iUp compte le nombre d'appels à la méthode up() ;
  • iUp++ ne reproduit pas la logique applicative (qui est une incrémentation modulo 10).

3.7 Mockito

Mockito en bref
C'est un framework de mocking (facilite le mocking).

a) Instancier une version mock d'une classe ou d'une interface (introspecte la classe d'origine) ;

	wheels[TENS] = mock(IThumbWheel.class);
  1. Mockito découvre les méthodes par introspection(1) ;
  2. Génère du byte code(2) à la volée pour générer une classe mock :
    • de même nom ;
    • avec des mêmes méthodes de même signature ;
    • autorisant les fonctionnalités de mocking du framework

b) Préparation d'un mock pour qu'une méthode retourne successivement certaines valeurs : 

	when(wheels[TENS].value()).thenReturn(8, 9, 0);

    => wheels[TENS].value() retournera 8 puis 9 puis 0 lors de la collaboration avec l'objet testé ;
    => équivalent à when(wheels[TENS].value()).thenReturn(8).thenReturn(9).thenReturn(0)

c) Vérification de l'état d'un mock après collaboration avec l'objet testé 

	verify(wheels[TENS], times(2)).up();

    => vérifie que la méthode up() a été appelée 2 fois sur wheels[1].

(1) Grâce à la bibliothèque de réflexion objenesis.
(2) Il utilise pour cela bytebuddy(3), une bibliothèque de génération et de manipulation de byte code.
(3) bytebuddy intègre elle-même la bibliothèque ASM de manipulation et d'analyse de byte code.
